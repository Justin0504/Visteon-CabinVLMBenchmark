# POI Information Retrieval — 设计方案（RAG / 工具调用）

> 为什么 POI 不能用"标注"解决，以及该用什么架构。补齐车外 8 类的最后一个缺口（以**方案**而非数据形式）。

## 1. 问题本质：答案不在像素里
POI 用例 = 看到建筑/地点 → 识别它是什么 → **给出关联信息**（名称、类型、营业时间、历史、周边）。
- 识别实体 = 视觉能做；
- **"它是哪个具体 POI + 它的信息" = 只存在于外部知识库/地图数据库，图里没有**。
- 所以让 VLM 直接"看图答 POI"必然幻觉 → **标注造不出正确数据**。这是**架构问题，不是标注问题**。

## 2. 目标架构：检索增强（RAG）+ 工具调用
```
图像 ──▶ VLM 实体识别（建筑/地标/店面文字）
              │（+ 可选：GPS/经纬度、OCR 出的招牌文字）
              ▼
         构造检索 query ──▶ 外部 POI API（高德/Google Places/维基）
                                   │ 返回候选 POI + 详情
                                   ▼
         VLM 整合检索结果 ──▶ 最终回答（带出处）
```
**核心理念**：模型学的不是"背下所有 POI"，而是 **①判断该题需要外部查询 ②生成正确的检索 query ③把返回结果整合成回答**。

## 3. 组件选型
| 组件 | 方案 |
|---|---|
| 实体/招牌识别 | 我们微调的座舱 VLM（已有）+ OCR（TextVQA 训练的读字能力）|
| 定位 | 车辆 GPS / 用户当前经纬度（PRD 输入里有定位信号）|
| **POI 检索 API** | 国内：**高德地图 POI 搜索/逆地理**；国际：**Google Places API**；通用知识：**Wikipedia API** |
| 整合 | VLM 把"识别结果 + API 返回"组织成自然语言回答 |

## 4. 怎么造训练数据（教"会查"而非"会背"）
不造"楼=某银行"的死记样本，而是造**工具调用样本**：
- **输入**：图（+ 模拟定位）+ 问题"这是什么地方/有什么"
- **目标输出（多轮/带工具）**：
  1. VLM：识别"这是一栋商业建筑，招牌写着 'XX Bank'" → **发起检索**：`poi_search(query="XX Bank", near=<gps>)`
  2. （注入）API 返回：`{name, category, rating, hours...}`
  3. VLM：基于返回生成回答"这是 XX 银行 X 路支行，营业至 18:00…"
- 这样模型学会**"识别→调用→整合"**的模式。可用少量人工 + 模板批量造。

## 5. 评测
- **检索触发准确率**：该查时是否发起查询（不该查时不乱查）；
- **query 质量**：生成的检索词是否命中正确 POI；
- **整合正确性**：最终回答是否忠于 API 返回（不额外幻觉）。

## 6. Capstone MVP 建议（务实）
完整 RAG 系统工作量大；终展 MVP 可做**两档**：
- **MVP-A（够交付）**：图 → VLM 识别招牌文字(OCR) → 调高德 POI 搜索 → 返回该店信息。覆盖"店面/商业 POI"这一最常见子集，**现场 demo 效果好**。
- **MVP-B（加分）**：加 GPS + 逆地理，支持"无招牌建筑"的 POI（靠定位而非文字）。

> 结论：POI 在数据集里**以"工具调用样本 + 架构"形式存在**，不是靠堆图标注。终展用 MVP-A 演示一个"看店招 → 查 POI → 回答"的闭环即可证明能力。
