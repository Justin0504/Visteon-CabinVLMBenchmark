# VLM/VLA 数据集客观评估方案(aojie)

> 对应任务:研究 VLA 评估机制、单列一个准确率指标、给出 finetune 提升的场景分类。
> 本方案在导师"序列对齐 + 回溯"框架基础上做分层修正,并用本项目已有实测数据作论据。

---

## 1. 一句话结论

**不同任务用不同评估算法,分层落地;先上"闭集 QA 准确率"作为主力可比指标,再叠加事实图匹配与推理-动作一致性。**

- 车外/座舱**场景描述** = 集合/图结构 → **图匹配**(顺序无关)
- **VLA 推理链(CoC/CoT)** = 有阶段因果 → **序列对齐**(导师方案适用于此,但不适用于场景描述)
- 所有任务 → 叠加一个**闭集 QA 准确率**作为可批量、可解释、跨模型可比的核心数字

---

## 2. 为什么不能只看语言相似度(本项目实测论据)

- **实测铁证**:对同一批 caption,LLM-judge 给两版打 **9.17 vs 9.29(几乎不区分)**;换成**冻结 VLM 在真值测试集上判**,立刻拉开 **6.45 vs 7.65**。
- 这独立复现了 DriveBench 的结论:**VLM 靠文本先验而非视觉 grounding**,语言流畅度 ≠ 视觉正确。
- 结论:**评估必须以"输出是否被客观证据(检测框/分类/属性/OCR/轨迹)支持"为准**,BLEU/CIDEr/单纯 LLM 打分都不可靠。

---

## 3. 分层评估指标

| 层 | 指标 | 客观证据 | 定位 |
|---|---|---|---|
| **L1 闭集 QA 准确率** | 计数/存在/灯色/标志类/方位 的 exact-match 准确率 | GT 标注程序化生成 QA | **主力·单列准确率指标(优先落地)** |
| **L2 事实图匹配** | object precision/recall、幻觉率、属性错、关系错 | bbox/分类/分割/OCR → Evidence Graph | 场景描述质量(顺序无关) |
| **L3 推理-动作一致性** | CoC 阶段对齐 + 决策是否被证据支持 + 动作是否与推理一致 | CoC 阶段 + 规则表 + ego 未来轨迹 | VLA 专用 |
| **L4 安全加权** | VRU/红灯/让行 高权重;建筑/天空 低权重 | 安全要素清单 | 贯穿 L1–L3 |
| **L5 校准/弃权** | 正确说"看不清/不确定"加分,自信错答重罚 | reject-QA | 安全场景必备 |

**L1 就是任务要的"单列准确率指标"**:从 GT 生成闭集 QA → 模型答 vs GT 答 → 一个干净的准确率,不依赖 LLM 主观、避开 ontology 对齐泥潭,作为 v1 上线。

---

## 4. 对导师方案的补充建议(我的 insight)

1. **序列对齐限定用于 VLA CoC**;场景描述以图匹配为主(顺序无关,"先说车还是先说灯"不该扣分)。
2. **直接用几何证据**(bbox IoU / 区域归属 / 距离 bucket / track ID),不要把一切先转自然语言再对齐。
3. **安全关键词用白名单 taxonomy + 混淆矩阵**判定(行人/骑车人/摩托/护栏),embedding 只用于非关键修饰词。
4. **总分必须分解**:幻觉(说了不存在)/ 漏述(GT有没说)/ 属性错 / 关系错 / 决策错,各出错误码 → 直接服务困难样本挖掘。
5. **加"推理-动作一致性"**(Alpamayo 核心):红灯+行人却输出"加速通过"应重罚。本项目可用 **nuScenes ego 未来位姿**做弱验证(decision 说减速 vs ego 实际是否减速)。
6. **加"元评测"**:这套打分**必须验证排序与人工偏好一致**,否则精心设计的分数可能与真实质量不相关——导师/审稿一定会问。
7. **加"防刷分"**:按 GT 事实打分会让模型背模板 → 必须配 **held-out + 对抗**(本项目已用印度/nuImages 未见集)。
8. **别一上来建统一 ontology**:nuScenes/Waymo/BDD/Mapillary 类别定义互不兼容,**先用单数据集(nuScenes)跑通再扩**。

## 5. 场景级评估(baseline vs finetuned,对应团队测试用例)

按 GT 给每帧打场景标签,问模型行驶动作,对照场景应有动作:

| 场景 | GT 来源 | 应有动作 |
|---|---|---|
| 正常通过 | 无 VRU 横穿 / 无红灯 | proceed |
| 行人过马路需减速 | VRU crossing=yes | slow / stop / yield |
| 前方红灯 | 提取灯状态=red | stop |
| 限速标志 | 标志类别 | 给出合规建议 |

→ 输出**每场景准确率 × 每模型**,直接得到"**finetune 在哪些场景提升明显**"。

## 6. Finetune 提升场景分类(本项目实测数据)

### 6.1 场景级决策(baseline v9 vs finetuned v15/v16)
| 场景 | v9 基线 | v15 | v16 | 解读 |
|---|---|---|---|---|
| 行人过马路→减速 | 95.0 | 95.0 | 90.0 | 都好(已饱和)|
| **正常通过→proceed** | **0.0** | **40.0** | **45.0** | **finetune 最大价值:基线病态过度保守(永远刹车),微调教会"安全时通过"** |
| 前方红灯→停 | 100.0 | 92.5 | 92.5 | 都好(已饱和)|
| **总体决策正确率** | 65.0 | **75.8** | **75.8** | **+10.8** |

### 6.2 闭集 QA 准确率(单列指标,exact-match)
| 类型 | v9 | v15 | v16 |
|---|---|---|---|
| 计数 | 86.7 | 85.0 | 86.7 |
| 骑车人存在 | 80.0 | 76.7 | 81.7 |
| 卡车存在 | 90.0 | 81.7 | 88.3 |
| 灯色 | 86.2 | **91.4** | 89.7 |
| **总准确率** | 85.7 | 83.6 | **86.6** |

### 6.3 finetune 提升场景分类(结论)
| 提升程度 | 场景类型 | 实测 |
|---|---|---|
| **最大** | 决策"何时通过"(纠正基线过度保守)| normal_pass 0→45% |
| **大** | 基座本来弱的专项任务 | VRU 分离 0→82.5%/87.5%、POI 检测 0→75%、灯色 86→91 |
| **中** | 有干净 GT 的细粒度识别 | 标志 95→100、景观 80→85 |
| **小/持平** | 基座本来就强的饱和感知 | 计数/存在(~86%)、通用场景描述(7.65→7.3)|

- **同数据集**:nuScenes frozen 测试集,v16 总准确率 86.6 ≥ v9 85.7;决策 +10.8
- **跨数据域**:印度未见集灯色 84.6%、nuImages 未见集 POI 检测 75% → 学到的是可迁移能力,非数据集记忆

## 7. 落地路线

- **v1(本周可出)**:L1 闭集 QA 准确率 + L3 场景决策一致性 → baseline vs finetuned 对比表(脚本已就绪:`eval_closedqa.py`、`eval_scenarios.py`)
- **v2**:L2 事实图匹配(单数据集 ontology)+ L4 安全加权 + L5 校准
- **v3**:元评测(与人工偏好相关性)+ 跨数据集统一 ontology
